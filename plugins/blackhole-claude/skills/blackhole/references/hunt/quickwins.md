# Hunt Kind: Quickwins

Scan heuristics, calibration table, and scoring rule for the `quickwins` hunt kind
(`kaizen.kinds`, `config-template.md`). Ported from `x-improve-hunt`'s dual-lens
structural + quality pass (ADR-006 § Hunt kinds).

## Scan heuristics

A `quickwins` wave looks for small, low-risk, high-confidence improvements — the kind of
finding a reviewer would flag as a WARN but that never rises to its own planned issue:

- **Dead code** — a function, variable, import, or export with no remaining external
  callers/references anywhere in the corpus.
- **Files >300 lines** — a single file exceeding 300 lines, usually a signal that it mixes
  more than one concern (see also `best-practices.md`'s SRP heuristic, which this overlaps
  with from the structural-size angle rather than the responsibility angle).
- **Nesting >4** — a function body with control-flow nesting (if/for/while/try) deeper than
  4 levels, hurting readability and testability.
- **Missing tests for existing logic** — a non-trivial exported function or class with zero
  test coverage, found by cross-referencing source files against the test suite (not a full
  coverage run — that is `coverage.md`'s job).
- **Over-abstraction** — a factory, strategy, wrapper, or interface with exactly one
  concrete consumer/implementation and no second one in sight (`V-YAGNI-03`).

Every finding is read-verified before it is reported: the hunter re-reads the cited
`file:line` and only reports `CONFIRMED` findings (`worker-schemas.md` § Hunter).

## Calibration table

| Heuristic | Trigger | Gain range | Effort range | Worked example |
|-----------|---------|------------|---------------|-----------------|
| Dead code | Unused function/variable/import with no external callers | 2–4 | 1–2 | Single unused private helper, no external callers → gain 3, effort 1 → Priority 3 × (11 − 1) = 3 × 10 = 30 (borderline, files at the floor) |
| Files >300 lines | A file exceeds 300 lines and mixes multiple concerns | 4–7 | 3–6 | 420-line module mixing parsing, validation, and I/O → gain 6, effort 4 → Priority 6 × (11 − 4) = 6 × 7 = 42 (moderate) |
| Nesting >4 | Function body nests control flow more than 4 levels deep | 3–5 | 1–3 | 5-level nested conditional in a request handler → gain 4, effort 2 → Priority 4 × (11 − 2) = 4 × 9 = 36 (borderline) |
| Missing tests for existing logic | Non-trivial exported function/class with zero test coverage | 5–8 | 2–4 | Untested payment-amount rounding helper, no test file references it → gain 7, effort 3 → Priority 7 × (11 − 3) = 7 × 8 = 56 (moderate) |
| Over-abstraction | Single-consumer interface/factory/strategy with no second implementation | 3–6 | 2–4 | Single-implementation factory wrapping one constructor call → gain 4, effort 2 → Priority 4 × (11 − 2) = 4 × 9 = 36 (borderline) |

`gain` and `effort` are each 1–10, matching the hunter output contract
(`worker-schemas.md` § Hunter, Finding shape). The ranges above are per-heuristic
calibration bands, not hard values — a hunter agent picks the specific score within the
listed range based on the concrete finding's actual scope (e.g. a 900-line file mixing
four concerns scores toward the top of the "files >300 lines" gain range, not the bottom).

## Scoring — V-PARETO-02 SSOT

`Priority = Gain * (11 - Effort)`; a finding must score `>= 30` to be filed as an issue, and
ready issues are sorted by Priority descending (`src/references/blackhole-vcodes.md`,
`V-PARETO-02`). This is the **only** scoring formula for the `quickwins` kind — no alternate
or per-kind formula is introduced. The calibration table above anchors this kind's
`gain`/`effort` inputs to the shared 1–10 scale; it does not replace the formula
(ADR-006 § Scoring model verdict: "the formula is sound and stays unchanged as the single
SSOT").
