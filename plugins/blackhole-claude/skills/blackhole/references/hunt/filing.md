# Hunt Filing Template

Shared issue-body template and priority bands used by the orchestrator when it files a
`CONFIRMED` hunt finding as a forge issue (ADR-006 § Orchestrator hunt dispatch). Every hunt
kind (`quickwins.md`, `best-practices.md`, `coverage.md`, and the not-yet-in-scope
`refactor`/`bug` kinds of #202) files through this one template — kinds do not each invent
their own issue-body shape.

## Issue-body template

Filed issues are titled `[Kaizen] <summary>` and use these fields, in this order:

| Field | Content |
|-------|---------|
| Summary | One-line description of the finding |
| Affected files | `file:line` reference(s) the finding is anchored to |
| Verbatim code | ≤ 8 lines, read-verified excerpt proving the finding is real (matches the hunter's `evidence_snippet`, `worker-schemas.md` § Hunter) |
| Rationale | Why this matters — see the root-cause-framing note below |
| Failure scenario | Concrete consequence if left unaddressed (a bug that recurs, a regression risk, a maintenance cost) |
| Fix direction | A recommended approach — not a full implementation, just enough for the planner to scope it |
| Footer | `gain`, `effort`, and the computed `Priority` (see § Scoring below) |

**Field-label reconciliation note**: the issue AC for this template (#198) spells this field
"Rationale" verbatim, while ADR-006 § Orchestrator hunt dispatch step 3 phrases the same slot
as "Root cause–rationale." This template uses the issue AC's literal label — it is the
acceptance criterion being graded — but the field's *content* still carries root-cause
framing, consistent with `V-FIX-01`'s "root cause, not symptom" convention already used
elsewhere in this campaign (`blackhole-vcodes.md`). This is a deliberate reconciliation, not
an oversight: a reviewer comparing this file against the ADR should read "Rationale" and
"Root cause–rationale" as the same field, worded per the issue's own text.

## Priority bands

| Priority | Band |
|----------|------|
| 80–100 | top priority |
| 60–79 | strong candidate |
| 40–59 | moderate |
| 30–39 | borderline (filed, lowest rank) |
| < 30 | archived, never filed |

These bands are a communication convention (wave notes, dashboard rows, ready-set sorting
labels) layered on top of the one scoring formula — they do not change the `>= 30` filing
floor.

## Scoring — V-PARETO-02 SSOT

`Priority = Gain * (11 - Effort)`; a finding must score `>= 30` to be filed as an issue, and
ready issues are sorted by Priority descending (`src/references/blackhole-vcodes.md`,
`V-PARETO-02`). This is the **only** scoring formula used by any hunt kind that files
through this template — no alternate or per-kind formula is introduced (ADR-006 § Scoring
model verdict: "the formula is sound and stays unchanged as the single SSOT... mercure's
mechanisms as input rules under the one formula, not as parallel formulas"). Findings scoring
below 30 are archived in the ledger and never reach this template.
